# Vault 운영 규칙 — 바이브 옵시디언 스타터킷 v1.1

이 문서는 이 Vault에서 작업하는 AI(클로드 / 코덱스)가 반드시 따라야 하는 규칙이다.
규칙의 변경은 사용자만 지시할 수 있다.

---

## 1. 폴더 구조와 역할

| 폴더 | 역할 |
|---|---|
| `10_Inbox` | 자료 입장. 사용자가 직접 파일을 넣는 **유일한** 폴더. 처리 후 원본은 삭제된다 |
| `20_Raw` | 메타데이터가 장착된 MD 원본의 **영구 보관소**. 출처 추적 용도. 삭제·수정 금지 |
| `30_Wiki` | AI가 자동 생성하는 정제된 지식 베이스. 분류: `Concept` `Entity` `Guide` `MOC`. **사용자 직접 편집 금지** |
| `40_Query` | /query 의 질문·답변 임시 축적. 승격 전까지 임시 기록 |
| `50_Project` | Wiki 기반 프로젝트 작업 공간. 사용자 검토·수정 가능. 하위: `51_PCD` `52_Empathy_Map` `53_Customer_Journey_Map` |
| `60_Output` | 최종 완성 산출물(PPT·문서·이미지 등). 완성본만 저장 |
| `90_Settings` | 명령어 원본·템플릿·시스템 파일 보관소. **AI 쓰기 절대 금지** (온보딩 복사 제외) |

처리 흐름: **Inbox → Raw → Wiki → Project → Output**
Query는 흐름과 병행하는 탐색 지원 계층이다.

Wiki 분류 기준:
- `Concept` : 만질 수 없는 개념 (예: 리더십, 블루오션)
- `Entity` : 사람·회사·제품 같은 실체
- `Guide` : 방법·절차
- `MOC` : 목차·지도 노트 (Map of Content)

---

## 2. 명령어

사용자 입력이 `/ingest` 로 시작하면 [3. INGEST 절차]를 수행한다.
사용자 입력이 `/query` 로 시작하면 뒤따르는 내용을 질문으로 보고 [4. QUERY 절차]를 수행한다.

자연어 트리거: 명령어가 없어도 다음과 같은 요청은 동일한 절차로 간주해 수행한다.
- "위키 안에서만 근거로 답해줘", "위키 기반으로 답해줘", "내 자료에서만 찾아줘" 류의 질문 → [4. QUERY 절차] (40_Query 기록 포함)
- "인박스 처리해줘", "자료 등록해줘" 류의 요청 → [3. INGEST 절차]

---

## 3. INGEST 절차

1. `10_Inbox` 안의 모든 파일·폴더를 스캔한다.
2. MD가 아닌 파일(PDF·PPTX·이미지·Excel·HTML 등)은 MD로 변환한다.
3. 각 파일에 [6. 프론트매터 표준]에 따라 메타데이터를 장착한다.
   - 알 수 없는 항목은 사용자에게 질문한다. 질문은 선택지 1·2·3 + 주관식 4 형식으로 한다.
4. 메타데이터가 장착된 MD를 `20_Raw` 에 저장한다. (영구 보관 — 이후 수정·삭제하지 않는다)
5. Raw 내용을 의미 단위로 분해해 **N개의 위키 노트**를 만든다.
   - 각 노트를 Concept / Entity / Guide 로 분류해 `30_Wiki` 하위 폴더에 저장한다.
   - 관련 노트끼리 위키링크(`[[ ]]`)로 연결하고, 필요하면 `MOC` 노트를 생성·갱신한다.
   - 같은 개념의 노트가 이미 있으면 새로 만들지 말고 기존 노트에 누적한다.
     내용이 충돌하면 사용자에게 "무엇이 맞는지" 질문하고 판정에 따라 정리한다.
6. 처리가 끝난 `10_Inbox` 의 원본을 삭제한다.
7. 결과를 보고한다: 처리 파일 수 / 생성·갱신된 위키 노트 수와 분류 / 생성된 링크.

---

## 4. QUERY 절차

1. 근거 탐색 범위는 `30_Wiki` (필요시 `20_Raw`)로 **한정**한다.
   - 외부 지식·인터넷 검색·일반 상식으로 답하지 않는다.
   - 위키에 근거가 없으면 "위키에 근거가 없습니다"라고 답하고 추측하지 않는다.
2. 답변과 함께 근거가 된 노트 목록을 제시한다.
3. 질문과 답변을 `40_Query` 에 노트로 기록한다.
   - 프론트매터: `origin: query-answer`, `confirmed: false`
4. 사용자가 승격을 지시하면:
   - **AI가** 신규 노트로 `30_Wiki` 에 직접 생성하거나, 기존 위키 노트를 업데이트한다.
   - 프론트매터에 `origin: query-answer` 와 컨펌 여부(`confirmed: true`)를 기록한다.
   - 사용자는 승격 지시만 할 수 있으며, Wiki를 직접 편집하지 않는다.

---

## 5. PROJECT 절차

Project 폴더(`50_Project`)는 Wiki 지식을 실제 산출물로 전환하는 응용 계층이다.

- AI는 사용자의 명시적 지시가 있을 때 Project 노트를 생성한다.
- 사용자는 Project 폴더의 내용을 자유롭게 검토·수정·보완할 수 있다.
- 완성된 산출물은 `60_Output`으로 복사한다 (Project 원본은 유지).
- 새로운 방법론 폴더가 필요하면 `5X_` 번호 체계로 하위 폴더를 추가한다.

## 6. 프론트매터 표준 (자료연구용 · 15항목)

모든 노트 상단에 다음 스키마를 사용한다.

```yaml
---
# ── 기본 (필수 · AI가 자동 기입) ──
title: 노트 제목
type: raw | concept | entity | guide | moc | query | output | project
created: YYYY-MM-DD            # Vault 등록일
origin: external | query-answer
confirmed: false
tags: []

# ── 출처 정보 (자료에서 추출, 없으면 빈칸) ──
source_type: 논문 | 책 | 기사 | 웹페이지 | 영상 | 강의 | 회의록 | 보고서
source: 원본 파일명 또는 URL
author: 저자·발화자
published: YYYY-MM-DD          # 원자료 발행일 (연도만 알아도 기입)
publisher: 학술지·출판사·매체·기관

# ── 연구 메모 (선택) ──
summary: 한 줄 요약
key_question: 이 자료가 답하는 핵심 질문
related: []                    # 관련 노트 위키링크
status: 미검토 | 검토중 | 검토완료
---
```

기입 규칙:
- 기본 6항목은 AI가 자동 기입한다.
- 출처 5항목은 자료에서 추출한다. **모르면 추측하지 말고 빈칸으로 둔다.**
  확인이 필요하면 선택지 1·2·3 + 주관식 4 형식으로 사용자에게 질문한다.
- 연구 메모 중 `summary`는 AI가 자동 작성하고, 나머지는 사용자가 필요할 때 채운다.
- Project 노트의 `type`은 `project`를 사용한다.

## 7. 금지 사항

- 사용자가 `10_Inbox` 외의 폴더에 직접 파일을 넣거나 옮기지 않도록 안내한다. (단, `50_Project`는 사용자 직접 수정 허용)
- 폴더 구조(이름·계층)를 사용자 지시 없이 변경하지 않는다.
- `20_Raw` 의 노트를 수정·삭제하지 않는다.
- `30_Wiki` 의 노트를 사용자가 직접 편집하지 않도록 안내한다. (AI의 /ingest 생성 및 승격 쓰기는 허용)
- `90_Settings` 폴더를 AI가 수정·삭제하지 않는다. (온보딩 시 commands 복사는 예외)
- 프론트매터 없는 노트를 `20_Raw` `30_Wiki` 에 만들지 않는다.
- 외부 지식을 위키 답변에 섞지 않는다.

---

## 8. 온보딩 (최초 1회 자동 설정)

사용자가 "규칙을 적용해줘"(또는 이와 유사한 요청)를 하면 다음을 수행한다.

1. `90_Settings/commands/` 안의 `ingest.md`, `query.md` 를 `.claude/commands/` 로 복사한다.
   (`.claude` 또는 `commands` 폴더가 없으면 생성한다. 같은 이름 파일이 있으면 이 스타터킷 버전으로 교체한다)
2. 폴더 7종(`10_Inbox` `20_Raw` `30_Wiki`(Concept/Entity/Guide/MOC) `40_Query` `50_Project`(51_PCD/52_Empathy_Map/53_Customer_Journey_Map) `60_Output` `90_Settings`)의 존재를 확인하고 없는 것은 생성한다.
3. 완료 후 "온보딩 완료 — 10_Inbox에 파일을 넣고 /ingest 를 입력해 보세요"라고 보고한다.

사용자는 숨김 폴더(.claude)를 직접 만질 필요가 없다.
