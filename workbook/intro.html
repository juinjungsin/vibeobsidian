바이브 옵시디언

나만의 PKM, 허들을 넘다

생성형 AI가 옵시디언 커뮤니티 플러그인에 연결되면서, 노동 집약적이던 PKM 구축이 "일단 해보는" 영역으로 바뀌었다. 복잡한 개념과 명령어는 잠시 미뤄두자. 내가 가진 자료로 그래프 뷰부터 만들어본다.
OVERVIEW

왜 지금, 바이브 옵시디언인가

PKM(개인지식관리체계)은 전문가들의 전유물이었다. 다양한 도구들이 존재했고, 사용자 분포는 대체로 다음과 같다.

PKM 사용자 분포

70% 20% 5% 5%

허들을 넘지 못한 70% · 넘었지만 유지 못 한 20% · 유지하지만 갈 길 잃은 5% · 프론티어 5%

강의의 목표 지점

프론티어 5%는 동경의 대상. 이 강의가 다루는 구간은 "허들 넘기"와 "유지하기"입니다.

Pain Point

흩어진 자료와 반복 설명의 비용을, 지식 관리 구조와 재사용 가능한 AI 맥락으로 해결한다.

문제현상교육에서 해결하는 방식
자료가 흩어짐PPT, PDF, HWP, 웹자료, 메모가 각자 따로 존재Obsidian Vault와 raw/wiki/output 구조로 통합
다시 찾기 어려움예전에 쓴 강의자료를 매번 다시 검색Wiki-link, Backlink, Search, Graph View로 재탐색
AI에 매번 설명해야 함강의 맥락, 고객사 맥락을 매번 프롬프트에 반복LLM Wiki를 통해 재사용 가능한 맥락 구축
자료 변환 품질이 낮음PDF/PPT를 요약하면 표·그림·출처가 사라짐원본 보존, sources 속성, 검수 체크리스트 적용
시각화가 부족함노트가 쌓여도 구조가 보이지 않음Graph View와 Canvas로 지식 구조 시각화

바이브 옵시디언 장단점

＋ 장점 기술적 타당성 높음

· 자동 링킹/백링크·인덱스 생성Ingest 시점에 지식 간 관계를 찾아 연결하므로, 사용자가 수동으로 링크를 만들지 않아도 지식 그래프 밀도가 올라간다.
· 3개 이상 토픽 교차 시 개념 페이지 생성반복 등장하는 개념을 wiki/concepts/로 분리하면 LLM Wiki의 합성 결과가 재사용 가능한 지식 단위가 된다.
· 그래프 뷰 임계점(200+ 노트) 초기 충족전통적 PKM은 노트가 쌓이기 전까지 효용을 체감하기 어렵다. 대량 raw를 빠르게 wiki로 전환하면 초기에 지식 지도를 볼 수 있다.
· 세션 간 컨텍스트 보존매번 100+ 파일을 다시 읽히지 않고 정제된 위키를 맥락 저장소로 쓰므로 토큰 비용과 반복 설명이 줄어든다.
· 지식의 복리 효과새 질문과 답변이 다시 위키에 누적되면서 다음 탐색의 출발점이 좋아진다.

－ 한계와 보완 운영 규칙 필요

· 위키 페이지 직접 편집 비권장재컴파일 시 LLM이 wiki 페이지를 다시 생성하면 사용자의 수동 편집과 충돌할 수 있다. 개인 메모는 별도 폴더에 두고 wiki-link로 연결하는 규칙이 필요하다.
· 인간의 거버넌스 레이어 필요Lint 규칙, 검수 체크리스트, sources 속성으로 자동 생성 결과를 관리해야 한다.
· 기술적 의존성Node 18+, Git, 포트 27124, MCP 서버 등은 입문자에게 진입 장벽이 될 수 있다. 설치 모듈, 원클릭 스크립트, 트러블슈팅 가이드가 전제되어야 한다.
· 외부 API 의존Claude/Codex API를 거치면 로컬 중심 데이터 소유권과 보안 민감도가 약해질 수 있다. 민감 자료는 로컬 처리와 제외 기준을 먼저 정해야 한다.
· 기존 자산 통합 방식Dataview, Templater, Zettelkasten 자산은 폴더·속성·링크 규칙을 정한 뒤 단계적으로 연결해야 한다.
