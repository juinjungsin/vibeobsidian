# 옵시디언 기반 LLM Wiki·PKM·RAG 실습형 강의 커리큘럼

작성일: 2026-06-10
근거 자료: 미팅 녹취록 3건, 김동준 박사 의견 문서 2건, 박종빈 소장 A~O 노트(옵시디언과정.pdf), 미팅 정리 PDF, 기존 커리큘럼 초안 2종
구조 원칙: 김동준 박사 의견(Raw→Distill→Wiki→Query→Output)을 뼈대로, 박종빈 소장 A~O 흐름을 실행 순서로 반영

표기 규칙: 자료에서 직접 확인된 내용은 표기 없음 / **[제안]** = 설계자 판단 / **[추정]** = 자료 판독·해석 개입

---

## 1. 전체 강의 콘셉트

이 강의는 옵시디언을 단순 메모앱이 아니라 AI가 읽고 쓸 수 있는 개인 지식베이스(LLM Wiki) 구축 도구로 사용하는 법을 가르치는 실습형 과정이다. 수강생은 강의 현장에서 Node.js·코덱스 CLI·옵시디언·클로디안을 직접 설치하고, 표준 폴더 구조(스타터킷)를 받아 자기 자료를 마크다운으로 변환해 Ingest하는 것부터 시작한다. 핵심 경험은 "질문할수록 자산이 쌓이는" 컴파운딩 순환으로, 일회성으로 소모되는 RAG·NotebookLM과의 차이를 실습으로 체감하게 한다. 김동준 박사의 Raw → Distill → Wiki → Query → Output 구조를 지식 흐름의 뼈대로 삼고, 박종빈 소장의 A~O 단계를 실행 순서로 배치했다. 포지셔닝은 Personal이 아닌 "프로젝트 Knowledge Management"로, 도메인 지식을 가진 팀장급 이상·스타트업 대표가 자기 자료로 PCD·MRD급 산출물을 만드는 것이 최종 목표다. 입문 과정(A~H)과 후속 과정(I~O, GraphRAG·Vault 결합·인사이트 도출)의 2단계로 분리 운영한다.

---

## 2. 핵심 학습 목표

수강 후 수강생은 다음을 할 수 있다.

1. Node.js(NPM)·코덱스 CLI·옵시디언·클로디안을 스스로 설치하고 Vault를 생성할 수 있다.
2. 스타터킷 폴더 구조 위에서 PDF·PPT·Excel 등 자기 자료를 MD로 변환해 Ingest할 수 있다.
3. 프론트매터·메타데이터가 붙은 노트가 위키(Concept·Entity·Guide·Map)로 분할·축적되는 구조를 설명할 수 있다.
4. Vault에 Query하고, 가치 있는 답변을 노트로 승격시켜 지식을 복리로 쌓을 수 있다.
5. 축적된 위키를 근거로 프로젝트 산출물(PCD·MRD 초안)을 생성할 수 있다.
6. RAG·NotebookLM·LLM Wiki의 차이를 구분하고, 자기 업무에 맞는 RAG 확장 방향을 설계할 수 있다.

## 3. 전체 커리큘럼 흐름

### 입문 과정 (도입 + A~H, 2일 × 4시간)

```
0. 도입 — 목적지·수단 비유, 담수화 비유, wow point(그래프 뷰) 시연
A. 사전 준비물 점검 — 계정(GPT 필수)·노트북(RAM 16GB 권장)
B. 설치 — Node.js(NPM) → 코덱스 CLI → 옵시디언 → 클로디안
C. Vault 생성 — 이름·경로 지정, 온보딩 준비
D. 온보딩 — 스타터킷(폴더 구조 자동 생성), 스킬 설치, 변환 파이프라인
E. 자료 등록(Ingest) — 클리핑·내 자료 → 프론트매터 → 의미 단위 분할 → Wiki
F. 질문과 답변(Query) — Q&A → 답변의 노트 승격 → 컴파운딩
G. 개념 정리 — RAG vs LLM Wiki vs NotebookLM 비교
H. 산출물 문서 흐름 — 위키 → PCD → MRD(→PRD) 프로젝트 적용
```

### 후속 과정 (I~O 재배열, 1일 4시간 **[제안]**)

```
I. GraphRAG — 노트 간 관계를 chunk로 활용하는 RAG 확장
J. 세밀화 — 위키가 정교해질수록 MD가 어떻게 달라지는가
L. Index(용어집) — A-B-C / ㄱㄴㄷ 색인 체계
K. Vault 공유 — 공통 사용, Sync, 오픈소스 분기
M. 그래프 Join — Vault 간 결합(갤럭시 비유), 조건부 시너지
N. 그래프 인사이트(View) — 링크·백링크 → Insight View → 공통분모 도출
O. 프레임워크 + Obsidian — 기존 산출물 × 프레임워크 결합 종합 실습
```

### 재배열 내역과 근거 **[제안]**

- A~H 순서는 원안 유지. 김동준 박사의 "선실습 후이론" 원칙과 박종빈 소장 원안의 순서가 이미 일치한다. 특히 G(개념 비교)를 실습(E·F) 뒤에 두는 원안 배치는 "철학·방법론은 마지막" 미팅 합의와 부합하므로 그대로 둔다.
- H는 김동준 박사의 PCD→MRD→PRD(Why→Who/What→What/How) 흐름과 결합해 "프로젝트 적용" 모듈로 확장했다. MRD까지만 다루는 것은 "MRD까지는 보안이 걸릴 일이 없다"는 김동준 박사 발언 근거.
- 후속 과정에서 L(Index)을 K(공유) 앞으로 이동했다. 원안 K→L→M은 'Vault 결합' 주제(K, M)가 색인(L)으로 단절된다. L은 개인 Vault 정비(J 세밀화)의 연장이므로 J 뒤에 붙이고, K→M→N을 "공유 → 결합 → 인사이트"의 연속 흐름으로 묶었다.
- Mothership–Satellite 2-Tier 구조는 입문에서 제외하고 후속 과정 K~M에 배치했다. "분리해서 하면 어려워하니 지금은 통합 구조, 나중에 고급 과정으로"라는 김동준 박사 발언 근거.

## 4. 세부 강의 모듈 (입문 과정)

### 모듈 0. 도입 (20분)

- **학습 목적**: 기대값 정렬과 동기 부여
- **핵심 개념**: 목적지–수단(자동차 vs 항공기), 담수화(원자료는 정제해야 마실 수 있다), 진행 원칙(일단 해본다, 강사와 동일 환경)
- **실습**: 없음. 강사의 완성 Vault 그래프 뷰 시연(wow point)
- **설치 도구**: 없음
- **예상 산출물**: 없음
- **막힐 지점**: 기대값 불일치("AI 만능론" 또는 "또 하나의 메모앱" 양극단)
- **강사 포인트**: 결과물을 먼저 보여준다. 개념 설명은 최소화. "내 자료를 넣어야 의미가 있다"(SWOT 비판 사례) 한 가지만 심는다.

### 모듈 A. 사전 준비물 (15분)

- **학습 목적**: 실습 환경 요건 점검
- **핵심 개념**: 유료 CLI 도구(코덱스), 계정과 결제의 사전 준비
- **실습**: 노트북 사양·계정·결제 상태 점검
- **설치 도구**: 없음(점검만)
- **예상 산출물**: 체크리스트 완료
- **막힐 지점**: 계정 ID/PW를 기억하지 못함 → 사전 공지에 "ID/PW 메모, 로그인/로그아웃 테스트" 포함
- **강사 포인트**: 프로그램 설치는 사전 과제로 내지 않는다("미리 깔아오라고 하면 난리 난다"). 계정 준비까지만.

### 모듈 B. 설치 (60~90분)

- **학습 목적**: CLI 환경과 옵시디언 구축
- **핵심 개념**: Node.js와 NPM의 관계, CLI 기초(관리자 권한, cd 명령)
- **실습** (NPM 우선 원칙):
  1. Node.js 공식 인스톨러 설치 → `node -v`로 확인 (Node.js 자체는 NPM 설치 불가, 공식 인스톨러가 표준)
  2. 코덱스 CLI: `npm install -g @openai/codex`
  3. (선택) 클로드 코드: `npm install -g @anthropic-ai/claude-code`
  4. 옵시디언: 공식 인스톨러 (NPM 미제공)
- **설치 도구**: Node.js, 코덱스 CLI, 옵시디언
- **예상 산출물**: 터미널에서 코덱스 실행 확인
- **막힐 지점**: 입문자가 가장 많이 막히는 구간. PowerShell 관리자 권한 미실행, Win/Mac 화면 차이, 설치 실패
- **강사 포인트**: OS별 화면 분리 준비, 시간 버퍼 최대 배정, 테이블(5명) 전원 클리어 후 다음 스텝 이동

### 모듈 C. Vault 생성 (15분)

- **학습 목적**: 첫 Vault 생성과 온보딩 준비
- **핵심 개념**: Vault '열기 vs 생성' 구분, 경로 개념
- **실습**: Vault 1개 생성(통일 규칙 경로), "신뢰하기" 처리
- **설치 도구**: 없음
- **예상 산출물**: 빈 Vault
- **막힐 지점**: 열기/생성 혼동, 압축 해제 위치, 폴더 중첩 생성 — 알파테스트에서 실증된 최다 오류 구간
- **강사 포인트**: 폴더 "안으로 들어가지 말고 폴더만 선택" 등 클릭 단위로 안내

### 모듈 D. 온보딩 — 스타터킷·스킬·변환 (70~90분)

- **학습 목적**: 표준 구조 자동 생성과 자료 변환 파이프라인 확보
- **핵심 개념**: 스타터킷(폴더 트리·폴더 간 규칙·자료 처리 방식 사전 규정), 번호 prefix 폴더("50번 프로젝트 들어가세요"), 스킬(슬래시 명령)
- **실습**: ① 클로디안 플러그인 설치(Vault를 연 후에만 가능) ② 스타터킷 적용 → "이 내용을 적용해줘" → 폴더 자동 생성 ③ 스킬 설치(Git 링크 → "스킬로 설치해줘") ④ PDF·PPTX·Image(OCR)·Excel·HTML → MD 변환 → 그래프 뷰 확인
- **설치 도구**: 클로디안, 옵시디언 스킬, 파일 변환 스킬(또는 MD 메이커)
- **예상 산출물**: 표준 폴더 구조가 생성된 Vault, MD 변환 파일 1~2개
- **막힐 지점**: 클로디안 설치 시점(Vault 이전에 설치 시도), MCP 타임아웃 등 베타 불안정성
- **강사 포인트**: "구조를 주고 건드리지 마"(에베레스트–베이스캠프 비유: 베이스캠프까지는 헬기로). 폴더 구조는 김동준 박사 안(Inbox / Raw / Wiki / Query / Project 최소 구성) 기반 **[추정: 최종 스타터킷 구조는 6/18 Draft 예정으로 자료상 미확정]**

### 모듈 E. 자료 등록 — Ingest (50분)

- **학습 목적**: 원본 1건이 의미 단위로 분할·분류 저장되는 과정 수행
- **핵심 개념**: Ingest(인박스 → Raw 이동 + 메타데이터·프론트매터 부착 + 의미 단위 분할), 1 대 다 분리(정유 공정 비유), Wiki 4분류(Concept·Entity·Guide·Map)
- **실습**: 웹클리퍼로 지정 링크(스테이지앤게이트·디자인씽킹·PCD·MRD·PRD 관련 9~10개) 클리핑 → `/ingest` → 분할 노트·프론트매터 확인
- **설치 도구**: 옵시디언 웹 클리퍼
- **예상 산출물**: 클리핑 자료가 분할된 위키 노트들
- **막힐 지점**: "파일을 넣으면 인제스트인가, 넣고 명령해야 하는가" 수준의 기본 혼동 — 실증된 질문
- **강사 포인트**: 핵심 규칙 — 인박스 투입 외 모든 조작은 클로디안 명령으로. 직접 폴더를 만지면 메타데이터 누락 → "쓰레기통이 된다"

### 모듈 F. 질문과 답변 — Query & Compounding (50분)

- **학습 목적**: 질문 → 답변 → 자산화 순환 체득
- **핵심 개념**: Query(위키 안에서만 답하게 가두기), 답변의 노트 승격(별도 노트 생성 또는 기존 노트 업데이트), 컴파운딩, 사람 컨펌(체크 시 가중치 상승)
- **실습**: 자기 Vault에 질문 3개 → 가치 있는 답변을 노트로 승격 → 그래프 뷰에서 연결 확인
- **설치 도구**: 없음
- **예상 산출물**: 승격된 Q&A 노트, 1일차 대비 성장한 그래프 뷰
- **막힐 지점**: 쿼리·위키·스킬 개념 구분 혼동
- **강사 포인트**: "안 되는 건 자연어로 클로디안에게 물어보면 된다" 상태까지가 목표 수준. "니가 알아서 해"가 제일 중요한 명령어

### 모듈 G. 개념 정리 — RAG vs LLM Wiki vs NotebookLM (40분)

- **학습 목적**: 실습으로 경험한 것의 원리 이해(소모형 vs 자산형)
- **핵심 개념**: RAG = 질문마다 처음부터 파싱(소모형), LLM Wiki = ingest 시점에 미리 합성 + 답변 누적(자산형), NotebookLM = 강력한 질의·요약이나 자산 누적 없음. 보안(로컬 보관)·비용·속도 차이. MD 영속성(옵시디언을 떠나도 자산이 남는다)
- **실습**: 같은 질문을 NotebookLM과 자기 Vault에 던져 결과·자산화 여부 비교
- **설치 도구**: 없음
- **예상 산출물**: 비교 경험
- **막힐 지점**: 용어 혼란(RAG, 파싱, 청킹) → 용어집 배포로 해소
- **강사 포인트**: 실습 후에 배치된 이유를 활용 — "방금 한 것이 바로 이것". NotebookLM 기능은 변동 가능하므로 강의 당일 최신 사양 재확인

### 모듈 H. 산출물 문서 흐름 — 프로젝트 적용 (60분)

- **학습 목적**: 위키 자산이 실무 기획 문서로 이어지는 흐름 체험
- **핵심 개념**: PCD(Why) → MRD(Who·What) → PRD(What·How), 프레임워크 위키가 미리 쌓여 있어야 산출물 품질이 달라짐(같은 입력 2페이지로 "밋밋한 GPT 문서" vs "PCD 16장")
- **실습**: 비즈니스 스토리 반 페이지 + E에서 쌓은 프레임워크 위키 → PCD 초안 생성
- **설치 도구**: 없음
- **예상 산출물**: PCD 초안 1부
- **막힐 지점**: 자기 자료(도메인 지식) 없이 따라만 하는 수강생 — 신청 시 비즈니스 스토리 제출로 사전 차단
- **강사 포인트**: MRD까지만(보안 이슈 회피). 옵시디언이 아니라 산출물이 목적임을 상기. 차기 과제: 2주간 자기 자료 수집·변환

## 5. 중복 내용 통합 결과

자료 전반(녹취 3건, 의견 문서 2건, PDF 정리본)에서 반복 등장해 하나로 통합한 메시지:

1. **옵시디언은 메모앱이 아니라 AI-ready 지식베이스 구축 도구다** — 노동집약(3년·300노트)에서 LLM 결합으로 진입장벽이 본질적으로 낮아졌다.
2. **내 자료를 넣어야 의미가 있다** — GPT 단독은 "정규분포 중간값" 답변, 내 자료 기반은 나에게 최적화된 끝단 답변. SWOT 비판 사례와 동일 논지.
3. **메타데이터·프론트매터·링크 구조가 검색과 그래프의 생명이다** — Ingest가 이를 보장하는 단계이고, 직접 조작하면 "쓰레기통"이 된다.
4. **정제된 데이터 조건은 가벼운 단서가 아니라 성패 변수다** — GIGO, 오염이 baked-in되면 미래 답변까지 오염.
5. **LLM Wiki는 자산형, RAG·NotebookLM은 소모형** — 컴파운딩이 본질. 답변이 다시 위키에 쌓인다.
6. **초보자 교육 원칙** — 선실습 후이론, 현장 설치, 스텝 바이 스텝, 스타터킷 제공, 용어 통제, 단일 도구(옵시디언 안에서 끝낸다).
7. **도구 표준은 코덱스 + 클로디안** — 클로드 코드는 비용 문제(작업 1건 20만원 사례)로 입문 비추천.
8. **디시플린의 외주** — 도구가 강제하지 않는 규율을 LLM이 대신 수행하는 것이 LLM Wiki의 차별점.

## 6. 제외한 내용

제외 기준별 유형 (개별 내용 생략):

- **일상·친분 대화**: 만남 인연 회고, 근황·성격 토크, 가족 농담, 식사·뒤풀이

- **사업 운영 사항**: 가격 책정, 모집 채널, 수익 배분, 서약서·페이백, B2B 영업 전략 — 커리큘럼 내용이 아닌 운영 사항이므로 제외 (강의 설계와 직결되는 일정·인원·대상 선별 기준만 반영)

- **브랜드 네이밍 논의(MOS/KOS/SOS/KMIOS)**: 김동준 박사 개인 시스템 명명 논의로, 커리큘럼 구조에는 SOS의 지식 흐름(Raw→Distill→Wiki→Output)만 반영하고 명칭 논의 자체는 제외

## 8. 강의 방향성 종합 의견

이하 전체가 자료 검토에 기반한 설계자 의견임 **[제안]**.

**초보자 난이도 관리.** 이 과정의 실패 지점은 개념이 아니라 설치와 클릭 단위 조작이다(알파테스트에서 Vault 열기/생성 혼동, 압축 해제 위치, 클로디안 설치 시점이 실제 막힌 지점으로 실증됨). 따라서 설명을 줄이는 것이 아니라 설명의 단위를 클릭 수준으로 낮추고, 테이블 전원 클리어 방식으로 낙오를 구조적으로 차단하는 것이 핵심이다. "유튜버처럼 좋은 점만 계속 설명하면 끝나고 뭘 해야 할지 모른다"는 피드백이 설계의 기준점이 되어야 한다.

**설치를 별도 단계로 분리해야 하는 이유.** 설치는 학습 내용이 아니라 학습의 전제 조건인데, 입문자에게는 이것이 가장 높은 벽이다. 본 학습(Ingest·Query)과 섞으면 설치 실패자가 이후 전부를 잃는다. 분리하면 시간 버퍼를 몰아줄 수 있고, "1일차에 환경이 완성된 상태"라는 명확한 중간 성취를 만든다.

**AI-ready 지식베이스로 설명하는 방법.** "메모앱 옵시디언"과의 차별은 기능이 아니라 구조로 설명해야 한다. 같은 자료라도 프론트매터·링크·의미 단위 분할이 있으면 AI가 읽을 수 있는 자산이 되고, 없으면 검색도 안 되는 파일 더미라는 대비를 실습으로 보여주는 것이 가장 효과적이다. MD 영속성("옵시디언을 떠나도 자산이 남는다")을 가치 제안으로 함께 묶는다.

**LLM Wiki와 RAG를 연결하는 교육 흐름.** 순서가 중요하다. RAG 개념을 먼저 가르치면 추상론이 되지만, 컴파운딩을 먼저 체험시킨 뒤 "방금 한 것이 RAG와 어떻게 다른가"로 비교하면 개념이 경험에 붙는다. G 모듈을 실습 뒤에 두는 원안 배치가 옳고, GraphRAG는 "연결에 관계 이름을 붙이면 답변 품질이 달라진다"는 한 문장으로 후속 과정의 동기를 만든다.

**프로젝트 실습 설계 방향.** 이 과정의 차별점은 "AI만으로도, 도메인 지식만으로도 안 된다"는 전제다. 따라서 실습 재료는 강사가 주는 샘플이 아니라 수강생 자신의 비즈니스 스토리여야 하며, 신청 단계에서 반 페이지 제출을 받아 도메인 보유자를 선별하는 운영과 맞물려야 한다. 프레임워크 위키를 미리 쌓는 단계(E)가 산출물 품질(H)을 결정한다는 인과를 과정 내내 반복 노출하는 것이 좋다.
